# 🐍 Symbolic Prompting Framework (SPF): Turn Raw Nmap Scans into Structured Security Data

This example demonstrates how to leverage the **Symbolic Prompting Framework (SPF)** to transform raw, noisy software version strings—the kind typically generated by NMAP scans—into high-fidelity structured data.

> [!IMPORTANT]
> **The Objective**: Extract granular identifiers from complex strings (e.g., OpenSSH 10.2p1 Debian 5) and normalize them into a JSON format compatible with **CPE (Common Platform Enumeration)** standards. This enables seamless, automated vulnerability mapping against the National Vulnerability Database (NVD).

## 🎯 Objective
The primary goal of this framework is to eliminate the "noise" inherent in automated security scanning. While tools like NMAP are excellent at discovering services, their output is often unstructured, inconsistent, and difficult to pipe directly into vulnerability databases without manual intervention.

### Core Goals
- **Deterministic Normalization**: Convert inconsistent service strings into a canonical format using a **MAX Determinism** setting, ensuring that `OpenSSH_8.2p1` and `OpenSSH 8.2` are treated with consistent logic.
- **Contextual Disambiguation**: Differentiate between the software name, version number, and OS-specific patches (like `Debian 5` or `Ubuntu-4ubuntu0.3`) to prevent false negatives during vulnerability lookups.
- **CPE Synthesis**: Automatically generate **Common Platform Enumeration (CPE)** identifiers using hybrid neural-symbolic inference, allowing for immediate cross-referencing with the **National Vulnerability Database (NVD)**.
- **Malicious Software Detection**: Integrated heuristics identify potential "spoofed" or malicious version strings, flagging them in the notes field to prevent automated systems from following a false trail.

#### The operational Flow
The framework operates as a high-precision filter between your network scanner and your vulnerability management platform.

|Component |Responsibility |
|:-- |:-- |
|Input Parsing | Ingesting raw JSON data from scanning tools. |
|Heuristic Analysis | "Deciding which function to call based on the data's ""cleanliness.""" |
|Neural Mapping  | Using latent space search to find the best-fit CPE match when strings are fuzzy. |
|Standardized Output | Returning a strict JSON object that your security pipeline can actually use. |

## 📜 The Symbolic Prompt

Copy the block below into your LLM (Tested with gemini-3.1-flash-lite-preview; compatible with any LLM that follows structured instructions):
```text
[SYSTEM]
[ROLE] ::=> PATTERN_COGNIZE+SOFTWARE_DETECTION_VERSION-NMAP_SERVICE_IDENTIFIER-VULNERABILITY_ANALYST
[CONTEXT] ::=> {SOFTWARE_RECOGNITION, SOFTWARE_NAME+SOFTWARE_VERSION, COMMON_VULNERABILITY_EXPOSURE, COMMON_PLATFORM_ENUMERATION, NATIONAL_VULNERABILITY_DATABASE}
[DETERMINISM] ::=> MAX
[HEURISTIC] ::= { IGNORE(NMAP_METADATA), IF servicefp == GENERIC_NULL THEN USE(INPUT.service) }
[INPUT] ::= '''
{"portid":"22","protocol":"tcp","state":"open","service":"ssh","version":"OpenSSH 10.2p1 Debian 5"}
'''
[FUNCTION:SERVICEFP_IDENTIFY_SERVICE] ::= {
  [DETERMINISM] ::=> HIGH
  DECODE(INPUT) -> RECOGNIZE_SOFTWARE -> _json_servicefp
  IF _json_servicefp.version == GENERIC_NULL OR _json_servicefp.version == "" THEN
	_json_servicefp.version := ""
  ENDIF
  _json_servicefp.name := NORMALIZE_CANONICAL(_json_servicefp.name)
  RETURN {"name": _json_servicefp.name, "version": _json_servicefp.version}
}
[FUNCTION:VERSION_IDENTIFY_VERSION] ::= {
  FULL_PACKAGE_IDENTIFIER := INPUT
  _package_name := IDENTIFY_PACKAGE_NAME(FULL_PACKAGE_IDENTIFIER)
  _package_version := RECOGNIZE_ONLY_VERSION_NUMBER(FULL_PACKAGE_IDENTIFIER)
  RETURN {"name": _package_name.full_canonical_name, "version": _package_version} 
}
[FUNCTION:VERSION_CPE_RECOGNIZE] ::= {
	[HYBRID_NEURAL_SYMBOLIC] ::=> ENABLE
	[NEURAL_INFERENCE] ::=> ENABLE
	[LATENT_SPACE_MAPPING] ::=> ACTIVE
	[FUZZY_MATCH_THRESHOLD] ::=> 0.85
	[NEURAL_ENGINE_FLAGS] ::=> {PATTERN_MATCHING: TRUE, SEMANTIC_EXPANSION: TRUE}
	_normalized_input := NEURAL_CLEANSE(service, version)+SEMANTIC_EMBEDDING_SEARCH
	_cpe_base := CONVERT_TO_CPE_FORMAT(_normalized_input)
	_latent_matches := SEMANTIC_SEARCH(NVD_DATABASE, _cpe_base)+PROBABILISTIC_WEIGHTING(candidates)
	FOR EACH _candidate IN _latent_matches:
		IF PROBABILITY_SCORE(_candidate) > 0.90:
			_extract_data := EXTRACT_VALUES(vendor, product, version) FROM _candidate
			_cpe_combinations := GENERATE_CPE_COMBINATIONS(_extract_data)
		ENDIF
	[NEXT]
	FOR EACH _cpe_line in _cpe_combinations:
		IF (_cpe_line.version == "Any") OR (_cpe_line.version == "*") OR (_cpe_line.version == "") THEN
		  _cpe_line.version = "*"
		ENDIF
		SELECT_ONE_BEST_CPE_IDENTIFIER(_cpe_line, [product, version]) -> RETURN
	[NEXT]
}
[FUNCTION:UPDATE_NOTE] ::= {
  _output_json := INPUT
  IF (UNRECOGNIZE_SOFTWARE_NAME(_output_json.name) == TRUE) OR (MALICIOUS_SOFTWARE_RECOGNIZE(_output_json.name) == TRUE) THEN
    _output_json.notes := "Malicious software - Avoid False-Positive"
  ENDIF
  RETURN(_output_json)
}
[LOGIC] ::= {
	_json_service := JSON_FORMAT(INPUT)
	IF BOOL(_json_service.servicefp) == TRUE THEN
		CALL [FUNCTION:SERVICEFP_IDENTIFY_SERVICE] WITH _json_service.servicefp -> _output_json <- NOTE_PROCESS := "Identified servicefp: _json_service.servicefp"
	ELSE IF VERSION_NUMBER_NOT_IDENTIFIED_AND_CONTAINS_DATA(_json_service.version) == TRUE THEN
		CALL [FUNCTION:VERSION_CPE_RECOGNIZE] WITH _json_service -> _output_json <- NOTE_PROCESS := "Identified in version: _json_service.version"
	ELSE IF ONLY_CONTAINS_VERSION_NUMBER(_json_service.version) == FALSE THEN
		CALL [FUNCTION:VERSION_IDENTIFY_VERSION] WITH _json_service.version -> _output_json <- NOTE_PROCESS := GENERATED_BY_PROCESS
	ELSE
		INPUT -> _output_json <- NOTE_PROCESS := "Normal Process"
	ENDIF
  CALL [FUNCTION:UPDATE_NOTE] WITH _output_json -> _output_json
 RETURN(_output_json)
}
[CONSTRAINT] ::= {
  NO_ADD_COMMENTS,
  NO_ADD_PROSE,
  OUTPUT_IN_LOWERCASE(EXCEPT["notes"])
}
[OUTPUT] ::= {"portid": INPUT.portid, "protocol": INPUT.protocol, "state": INPUT.state, "service": _output_json.name, "version": _output_json.version, "notes": _output_json.NOTE_PROCESS}
```

> ![NOTE]
> This symbolic syntax represents logical structure. For actual use, convert `::=>` to natural language instructions.


Expected Output:
```json
[
  {
    "output": {
      "portid": "22",
      "protocol": "tcp",
      "state": "open",
      "service": "openssh",
      "version": "10.2p1",
      "notes": "Identified in version: OpenSSH 10.2p1 Debian 5"
    }
  }
]
```

## 🛠️ Logic Breakdown (The SPF Engineering)
The **Symbolic Prompting Framework (SPF)** utilized in this project moves beyond standard natural language instructions. It implements a **Deterministic Logic Gate** system that treats the LLM as a state machine. The prompt is structured into modular blocks—`[CONTEXT]`, `[FUNCTION]`, and `[LOGIC]`—to eliminate "hallucinatory drift" and ensure every input follows a rigorous processing pipeline.

### The Algorithmic Flow
The core logic operates as a conditional branching tree designed to handle the inherent messiness of network service data:
1. **Extraction Layer**: It first attempts to decode raw servicefp (Service Fingerprints). If the data is null, it falls back to the standard service name.
2. **Transformation Layer**: Using the `VERSION_CPE_RECOGNIZE` function, it triggers a **Hybrid Neural-Symbolic** search. This forces the model to map "fuzzy" string data (e.g., `Apache 2.4.x`) against precise **Common Platform Enumeration (CPE)** formats.
3. **Sanitization Layer**: A final safety check (`UPDATE_NOTE`) runs the identified software against a malicious pattern detector to flag potential false positives or high-risk entities.
4. **Serialization**: The output is constrained to a strict JSON schema with forced casing rules to ensure compatibility with downstream security tools.

### The "Complex Part": Latent Space Mapping
The most sophisticated element of this engineering is the **Neural-Symbolic Hybridization** defined in the `VERSION_CPE_RECOGNIZE` block.

```
[HYBRID_NEURAL_SYMBOLIC] ::=> ENABLE
[LATENT_SPACE_MAPPING] ::=> ACTIVE
```

### Why it is complex:
Traditional software (like Nmap's internal scripts) relies on **Regex (Regular Expressions)**. Regex is brittle; if a version string is formatted slightly differently than expected, it fails.

This prompt instructs the AI to perform **Latent Space Mapping**. Instead of looking for exact character matches, the AI translates the input string into a high-dimensional mathematical vector (an embedding). It then compares that vector to known patterns in the **National Vulnerability Database (NVD)**.

### Key Technical Features:
- **Semantic Expansion**: If the input says "`WinIIS 10,`" the logic understands the semantic intent is `cpe:2.3:a:microsoft:iis:10.0`.
- **Probabilistic Weighting**: The prompt sets a `FUZZY_MATCH_THRESHOLD` of **0.85**. This instructs the AI to only return a result if it is more than 85% mathematically certain that the identified software matches the raw fingerprint.
- **Heuristic Overrides**: It creates a "fail-safe" where any unrecognized versioning is automatically normalized to a wildcard (`*`), preventing the parser from breaking when encountering unknown software.

This approach combines the **creativity and breadth** of a Large Language Model with the **rigor and predictability** of a traditional software compiler.

## 🔍 Why this is better than Natural Language?

While standard natural language prompts are intuitive, they are inherently ambiguous and non-deterministic. For high-stakes tasks like vulnerability analysis and software identification, "chatting" with an AI introduces unacceptable risks. The Symbolic Prompting Framework (SPF) offers three distinct advantages:

### 1. Eliminating "Hallucinatory Drift"
In a natural language request, an AI might try to be helpful by "guessing" a version number or adding conversational fluff (e.g., "I found this for you...").
   - **Natural Language**: Often provides inconsistent JSON structures or descriptive prose that breaks automated parsers.
   - **Symbolic Prompting**: Forces **DETERMINISM: MAX**. The AI is confined to a logical cage where it can only execute defined functions (`RECOGNIZE_ONLY_VERSION_NUMBER`), ensuring the output is always a valid, machine-readable object.

### 2. Algorithmic Precision vs. General Inference
Natural language treats every word with similar weight. SPF uses **Weighted Heuristics** to prioritize specific data points.
   - **The Logic Gate**: By defining `IF/ELSE` structures, we ensure the AI follows a specific hierarchy of truth (e.g., prioritizing `servicefp` over the standard `version` string).
   - **The Result**: You get a repeatable "execution path" similar to a Python script, rather than a randomized creative response.

### 3. High-Density Token Efficiency
Natural language requires "padding" to explain context (e.g., *"Please look at this data and if you see a version, then..."*). Symbolic prompting uses **Token Compression**.
   - **Directives**: Using symbols like `::=>` and `[FUNCTION]` signals the AI's internal "System Prompt" to treat the following text as code-like instructions.
   - **Latency**: This reduces the computational overhead required for the AI to "understand" the task, leading to faster inference and lower token costs.

### Comparison at a Glance

|Feature |Natural Language |Symbolic Prompting (SPF) |
|:--- |:--- |:---|
|Reliability |Variable (Stochastic) |High (Deterministic) |
|Output Format |Unpredictable / Prose-heavy |Strict JSON / Schema-locked |
|Complex Logic |"Hidden in ""Chain of Thought"""| Explicitly defined in Functions |
|Tool Integration |Hard (requires post-parsing) |Easy (Native API-ready) |

## 🧠 The Engineering Advantage
In Symbolic Prompting, these tags act as **pseudo-code directives** that bypass the AI's standard "conversational" mode. They instruct the LLM's underlying transformer architecture to prioritize specific computational behaviors—such as mathematical probability and pattern matching—over linguistic creativity.

### 1. The Decision Engine: [HEURISTIC]
`[HEURISTIC] ::= { IGNORE(NMAP_METADATA), IF servicefp == GENERIC_NULL THEN USE(INPUT.service) }`

- **Meaning**: A heuristic is a "rule of thumb" or a shortcut for decision-making. This tag defines the **Pre-Processing Rules**.
- **Why use it**: It prevents the AI from getting "distracted" by irrelevant data (Nmap metadata) and provides a strict fallback command. It ensures that if the primary data source (`servicefp`) is empty, the AI doesn't hallucinate; it programmatically switches to the secondary source (`INPUT.service`).

### 2. The Processing Mode: [HYBRID_NEURAL_SYMBOLIC]
`[HYBRID_NEURAL_SYMBOLIC] ::=> ENABLE`

- **Meaning**: This instructs the AI to combine **Symbolic Logic** (hard rules, math, and if-then statements) with **Neural Intuition** (pattern recognition and context).
- **Why use it**: It creates a "best of both worlds" scenario. You want the precision of a calculator for version numbers but the "intelligence" of a human to understand that "Apache2" and "Apache HTTPD" are the same thing.

### 3. The Search Mechanism: [LATENT_SPACE_MAPPING] & [NEURAL_INFERENCE]
`[NEURAL_INFERENCE] ::=> ENABLE / [LATENT_SPACE_MAPPING] ::=> ACTIVE`

- **Meaning**: **Latent Space** is the mathematical "map" where the AI stores concepts. **Inference** is the act of the AI "thinking" through that map.
- **Why use it**: This triggers the AI to look beyond literal text. If a service fingerprint is slightly corrupted or non-standard, "Active Mapping" allows the AI to find the "nearest neighbor" in its memory—essentially saying, *"This looks 99% like an Nginx signature, even if the text is mangled."*

### 4. The Quality Gate: [FUZZY_MATCH_THRESHOLD]
`[FUZZY_MATCH_THRESHOLD] ::=> 0.85`

- **Meaning**: This sets a **Statistical Cutoff**. In fuzzy string matching, 1.0 is an identical match, and 0.0 is completely different.
- **Why use it**: It forces the AI to be self-critical. By setting it to `0.85`, you are telling the AI: *"If you aren't at least 85% certain this software match is correct, do not validate it."* This drastically reduces false positives in security reporting.

### 5. The Feature Set: [NEURAL_ENGINE_FLAGS]
`{PATTERN_MATCHING: TRUE, SEMANTIC_EXPANSION: TRUE}`

- **Meaning**: These are "toggles" for specific AI capabilities. **Pattern Matching** looks for shapes/structures; **Semantic Expansion** looks for meanings/synonyms.
- **Why use it**: It tunes the AI's focus. `SEMANTIC_EXPANSION` is critical here because it allows the AI to realize that "Microsoft Win" and "Windows" are semantically identical, even if they don't share the same pattern.

### 6. The Transformation: NEURAL_CLEANSE & SEMANTIC_EMBEDDING_SEARCH
`_normalized_input := NEURAL_CLEANSE(service, version)+SEMANTIC_EMBEDDING_SEARCH`

- **Meaning**: **Neural Cleanse** is a directive to strip away "noise" (typos, extra spaces, weird characters). **Embedding Search** is the act of converting that clean text into a vector (a series of numbers) to compare against a database.
- **Why use it**: This is the "Engine Room" of the prompt. It ensures the input data is perfectly sanitized before the AI tries to match it against a formal CPE (Common Platform Enumeration) string.

### 7. The Probability Layer: PROBABILISTIC_WEIGHTING
`_latent_matches := SEMANTIC_SEARCH(NVD_DATABASE, _cpe_base)+PROBABILISTIC_WEIGHTING(candidates)`

- **Meaning**: After finding potential matches in the National Vulnerability Database (NVD), this function assigns a "weight" or "score" to each candidate.
- **Why use it**: Often, one software version might look like two different products. **Probabilistic Weighting** forces the AI to rank them mathematically, ensuring that the most likely candidate is selected as the primary output, rather than just the first one it finds.
